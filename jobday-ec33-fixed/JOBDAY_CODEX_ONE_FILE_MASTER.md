## Source of truth
- The repository code is the truth about what currently exists.
- The two numbered documents above are the truth about the target product and UX direction.
- When code and documents conflict, report the conflict. Do not silently choose or rewrite large areas.
- Do not treat plans, mockups, inaccessible URLs or unrun tests as completed work.

## Scope control
- Do not build the entire platform in one task.
- The first task is audit and planning only unless the user explicitly approves prototype implementation.
- Keep Community, Jobs, Shop, Ads, Membership and native payments behind feature flags until their phase is approved.
- Preserve useful existing code and patterns. Do not rewrite Auth, database schema or routing without an approved migration plan.

## Product rules
- `Occupation` is the common organizing object.
- The user-facing journey is: listen/watch → explore an occupation → ask or find a job.
- Community, commerce and advertising must appear in relevant occupational context, not as disconnected portal sections.
- The product must feel like a polished, human-centered editorial media platform, not an industrial or government job board.
- Use real Korean content lengths; do not use lorem ipsum, fake metrics or fake reviews.

## Trust, privacy and security
- Protect sensitive data with server authorization and Supabase RLS.
- Store identity evidence, resumes, appearance consent forms, private contacts and report evidence privately.
- AI may assist with drafting, classification and risk flags, but may not independently publish content, verify employers, permanently ban users or make legal decisions.
- Log important moderation, verification, role, advertising and publishing actions.
- Do not expose unverified employers, public blacklists, manual recruitment matching, success fees or sensitive status information.

## Quality
- Every implementation plan must state scope, exclusions, permissions, states, analytics, tests, browser verification and rollback.
- Test mobile widths 360, 390 and 430px as well as desktop.
- Run the repository's actual lint, typecheck, test and build commands. Report failures exactly.
- Do not claim completion without evidence.

# 3. 시장 기회와 수요 가설

콘텐츠 측면에서 긴 대화와 팟캐스트는 이미 대규모 소비 습관입니다. YouTube는 2025년 1월 기준 팟캐스트 콘텐츠의 월간 활성 시청자가 10억 명을 넘었다고 발표했습니다\[S1\]. 이는 JOBDAY가 오디오 전용 앱에 갇히지 않고 영상·오디오를 동시에 제공해야 하는 근거입니다.

직업 이야기의 소재도 넓습니다. 2025년 말 한국의 취업자격 체류외국인은 594,047명이고 전체 체류외국인은 2,783,247명입니다\[S2\]. 외국인 노동자, 해외 취업자, 교대근무자, 기술직, 자영업자, 서비스직 등 기존 채용 플랫폼에서 충분히 설명되지 않는 삶의 세계가 큽니다.

# 6. 모듈별 제품 정의

## 6.1 JOBDAY MEDIA

- 40~70분 영상 팟캐스트를 원본으로 제작하고, 오디오 피드·쇼츠·글 인터뷰로 파생한다.

- 직업정보만 묻지 않고 시작 계기, 초보 시절, 현장 문화, 사람, 돈, 생활 리듬, 그만둔 이유와 남은 것을 다룬다.

- 유명인보다 실제 종사자와 유경험자를 주인공으로 삼는다.

- 밤에 듣는 잔잔한 “야간편”과 낮에 보기 좋은 편집형 “현장편”을 구분한다.

## 6.2 JOBDAY COMMUNITY

- 직종·지역·주제별 게시판과 에피소드 후속 대화를 제공한다.

- 실명 회사 폭로가 아니라 직업 경험, 기술, 질문, 준비방법 중심으로 운영한다.

- 현직 인증은 공개 배지가 아니라 운영자 비공개 검증을 기본으로 한다.

- 익명성은 허용하되 계정·신고·평판 기록을 통해 무책임한 게시를 줄인다.

## 6.3 JOBDAY JOBS

- 초기에는 직업정보제공사업 형태로 공고 정보를 제공하고 사용자가 직접 지원하게 한다.

- 플랫폼이 특정 후보자를 골라 기업과 연결하거나 채용 성공 수수료를 받는 단계는 관련 등록을 마친 뒤에만 진행한다.

- 공고는 급여, 지급주기, 시간, 장소, 업무, 안전장비, 숙소, 계약형태를 표준 필드로 작성한다.

- 콘텐츠와 직업 페이지에서 공고로 자연스럽게 이동한다.

## 6.4 JOBDAY SHOP

- 일반 종합몰이 아니라 “직업인이 실제 쓰는 것”을 선별하는 편집형 커머스다.

- 첫 단계는 제휴링크·위탁·소량 사입으로 운영하며 자체 창고와 오픈마켓 입점을 받지 않는다.

- 에피소드 출연자의 사용 경험, 직종별 스타터 키트, 안전·편의 개선 상품을 중심으로 한다.

- 상품 수보다 추천 근거와 직업 맥락을 우선한다.

## 6.5 JOBDAY BUSINESS

- 기업 공고, 상단 노출, 직종 카테고리 스폰서, 기업 직무 소개 콘텐츠, 채용 브랜딩을 판매한다.

- 모든 유료 노출은 “광고/협찬”을 명확히 표시하고 일반 출연과 편집권을 분리한다.

- 무관한 배너를 대량 판매하지 않고 직업·지역·에피소드 문맥에 맞는 광고만 허용한다.

# 24. 제품개발 원칙

- Mobile first: 현장 종사자와 이동 중 청취자를 기준으로 모바일에서 먼저 완성한다.

- Build core, buy commodity: 직업 그래프·커뮤니티·신뢰 데이터는 직접 만들고 영상호스팅·결제·배송은 외부 서비스를 활용한다.

- One identity: 콘텐츠, 커뮤니티, 공고, 샵이 하나의 계정과 직업 팔로우를 공유한다.

- Feature flags: 미검증 모듈은 코드가 있어도 일반 사용자에게 노출하지 않는다.

- No microservices initially: Next.js와 Supabase 중심의 단순 구조를 유지한다.

- AI assists, humans decide: 콘텐츠·공고·신고 검토에서 AI는 분류와 초안을 만들고 최종 조치는 사람이 한다.

- Privacy by default: 경력증빙·연락처·지원서·출연동의는 공개 데이터와 분리한다.

# 28. 핵심 사용자 여정

1\. 청취자: 쇼츠/검색 → 에피소드 → 직업 허브 → 직업 팔로우 → 커뮤니티 → 관련 공고 저장.

2\. 현직자: 에피소드 청취 → “나도 출연/경험 공유” → 비공개 검증 → 출연 또는 글 → 직업 기여자 배지.

3\. 구직자: 직업 탐색 → 실제 경험 청취 → 조건 필터 → 기업검증 공고 → 지원 → 지원 상태 확인.

4\. 기업: 기업회원 → 사업자 검증 → 공고 작성 → 검수 → 공개 → 성과 확인 → 추천노출 결제/갱신.

5\. 구매자: 에피소드/직업 페이지 → 출연자 추천 상품 → 근거/대안 확인 → 외부 또는 내부 결제.

6\. 광고주: 기업서비스 문의 → 직종/목표 정의 → 광고 심사 → 계약/표시 → 캠페인 → 성과보고.

# 31. 기능 요구사항: MEDIA

- 에피소드 CRUD, 초안/검수/예약/공개/비공개 상태.

- YouTube·오디오 URL, 타임스탬프, 원고, 자막, 출처, 관련 직업·게스트·스폰서 연결.

- 가명/얼굴블러/음성변조 여부와 동의 버전을 관리자만 확인.

- 에피소드 페이지에서 재생, 저장, 직업 팔로우, 댓글, 관련 공고·상품 이동.

- 클립별 플랫폼·성과 기록, 원본 에피소드로 UTM 연결.

- 스폰서가 있으면 제목 근처와 플레이어 앞에 명확한 광고 표시.

# 32. 기능 요구사항: COMMUNITY

- 보드별 규칙, 직업/지역 태그, 최신/추천/미답변 필터.

- 신규 계정은 첫 72시간 외부링크·전화번호·회사명 패턴 게시 제한.

- 작성자가 익명 표시를 선택해도 내부 profile_id는 유지.

- 신고 유형: 개인정보, 비방, 허위정보, 불법구인, 혐오, 스팸, 안전위험.

- 숨김·삭제·경고·기간정지·영구정지와 이의제기 기록.

- 직업 질문에 verified_worker의 답변을 표시하되 전문자격을 자동 보증하지 않는다.

# 33. 기능 요구사항: JOBS

- 기업회원 가입 → 회사 생성 → 사업자/담당자 검증 → 공고 초안 → 자동 필수값 검사 → 사람 검수 → 게시.

- 공고 만료, 재게시, 중복 탐지, 연락처/링크 위험 검사.

- 구직자는 저장·알림·지원·지원내역 확인.

- 초기 추천은 사용자의 선택한 직업/지역/형태에 따른 노출이며 운영자의 개별 알선 기능은 만들지 않는다.

- 유료 추천 노출은 표시하고 일반 공고와 동일한 품질 검수를 적용.

- 허위공고 신고 시 즉시 검토대기 및 노출 제한 옵션.

# 34. 기능 요구사항: SHOP·ADS

- 상품은 occupation_id 또는 episode_id 중 하나 이상과 반드시 연결.

- commerce_mode는 affiliate, external_store, direct 세 가지로 시작.

- 외부링크 클릭과 구매전환을 가능한 범위에서 추적하며 수수료 관계 표시.

- 캠페인은 직업·지역·콘텐츠 시리즈 타기팅만 허용하고 민감정보 기반 타기팅 금지.

- 광고 관리자 화면에 계약기간, 표시문구, 소재승인, 노출/클릭, 종료 상태 제공.

- 콘텐츠형 광고에는 제작진 편집 독립과 광고주 검토 가능 범위를 기록.

# 39. Definition of Done와 테스트

- 기능은 UI가 보이는 것으로 끝나지 않고 정상·권한없음·삭제/만료·중복·오류 상태를 테스트한다.

- 모든 민감 테이블은 RLS 자동테스트가 있고 익명/다른 사용자/기업/관리자 시나리오를 검증한다.

- 공고와 에피소드 게시 전 필수 검수 상태가 없으면 서버에서 차단한다.

- 광고/제휴 콘텐츠는 disclosure_text가 없으면 publish할 수 없다.

- 동의가 철회된 게스트의 공개 상태를 일괄 변경할 수 있다.

- 모바일 360px 폭에서 핵심 페이지가 가로 스크롤 없이 동작한다.

- 한국어 입력·검색·줄바꿈·날짜/금액 표시를 검증한다.

- 접근성: 키보드 탐색, 포커스, 대체텍스트, 명확한 오류메시지.

- 배포 전 DB 백업·복원, 파일 접근, 관리자 계정 MFA와 감사로그를 확인한다.

- 출연료·교통비·식비·익명화 비용을 기록하며 무료 출연에만 의존하지 않는다.

- 섭외 거절 이유와 공개 후 불편 신고를 기록해 출연 제안을 개선한다.

- 콘텐츠 다양성은 토큰 채우기가 아니라 직업과 삶의 세계를 정확히 보여주는 편성 원칙으로 관리한다.

# 67. 검색·추천·알림의 신뢰 원칙

무엇을 먼저 보여주는지가 JOBDAY의 편집 철학입니다. 돈과 자극이 추천을 지배하면 직업인의 신뢰를 잃습니다.

- 유료 노출은 명확히 표시하고 일반 추천 점수와 별도 슬롯으로 관리한다.

- 완주율만 최적화해 충격·갈등 콘텐츠가 과도하게 상위화되지 않도록 신고·다양성·만족 신호를 함께 본다.

- 새로운 직업과 신인 출연자에게 탐색 노출을 제공하고 한 직업군·성별·국적에 편중되는지 정기 감사한다.

- 사용자는 추천 이유, 관심 직업 변경, 기록 초기화, 알림 빈도를 이해하고 조정할 수 있다.

- 검색은 직업명뿐 아니라 별칭·현장용어·사람·근무환경·자격·에피소드 대본·공고를 통합한다.

- 팔로우 직업 새 이야기, 이어듣기, 질문 답변, 공고 알림은 별도로 켜고 끌 수 있으며 야간 방해를 제한한다.

| **추천 목표**    | **사용 신호**                        | **보호장치**                        |
|------------------|--------------------------------------|-------------------------------------|
| 좋은 이야기 발견 | 완주, 저장, 공유, 후속 직업 탐색     | 낚시 클릭만 높은 콘텐츠 감점        |
| 직업 이해        | 직업 허브 체류, 질문 열람, 출처 확인 | 불명확한 급여·정보에 신뢰 표시 금지 |
| 안전한 일자리    | 검증 필드, 저장, 지원 완료           | 광고비로 검증 우회 금지             |
| 다양한 세계 발견 | 새 직업 탐색과 균형                  | 민감정보 기반 타기팅 금지           |

| 출연 동의서·원본 | 제작·법무 최소권한    | 계약·게시 기간과 분쟁 가능성 고려 | 철회 시 공개물 처리와 원본 보관 근거 분리 |
| 지원서·연락처    | 지원자·해당 기업 제한 | 채용 절차와 동의 기간             | 지원 철회·계정 삭제 절차                  |
| 신고 증거        | 운영·법무 제한        | 이의제기·분쟁 기간                | 목적 종료 후 파기                         |
| 행동 분석        | 개인식별 최소화       | 제품 개선에 필요한 기간           | 동의·정책에 따른 삭제/익명화              |
| 감사 로그        | 보안·운영 제한        | 중요 행위 추적 기간               | 무단 변경 금지, 정해진 만료               |

- 실제 보유기간은 목적·법률·분쟁 가능성을 검토해 개인정보 처리방침과 내부 관리계획에 동일하게 반영한다.

- AI 도구로 민감 파일을 전송하지 않으며 필요한 경우 위탁·국외이전·학습 활용 여부를 검토한다.

- 프로덕션 데이터를 개발·프롬프트·스크린샷에 무단 복사하지 않는다.

- 분석 이벤트에는 주민번호, 여권, 체류자격, 전화번호, 이력서 본문을 넣지 않는다.

3. 시각 디자인 방향
────────────────────────────────────────

디자인의 중심은 “직업”이 아니라 “그 일을 살아가는 사람”입니다.

사진 원칙:

- 실제 인물의 표정, 손, 작업환경, 일상적 순간을 중심으로 사용
- 과도하게 연출된 스톡사진처럼 보이지 않게 함
- 사무직, 서비스직, 기술직, 예술직, 공공직, 자영업, 플랫폼 노동,
  야간·교대직, 외국인 근로자까지 균형 있게 표현
- 특정 성별, 나이, 국적, 육체노동 이미지에 치우치지 않음
- 익명 출연자의 경우 정체성을 보호하면서도 사람의 온기를 살리는 촬영 방식 사용

권장 시각 톤:

- Editorial media와 현대적 라이프스타일 플랫폼의 중간
- 충분한 여백
- 강한 타이포그래피 위계
- 인물 중심의 큰 썸네일
- 콘텐츠가 먼저 보이고 UI 장식은 뒤로 물러남
- 밝은 배경과 깊은 잉크 계열의 대비
- 포인트색은 신뢰와 생동감을 주되 경고색처럼 보이지 않게 사용
- 한 화면에서 핵심 강조색은 1개, 보조 강조색은 최대 1개
- 아이콘은 기능 보조용으로만 사용
- 의미 없는 이모지 사용 금지

초기 권장 색상 토큰을 제안하되,
현재 브랜드 자산을 감사한 후 최종 선택하십시오.

- 제목은 개성과 밀도를 가지되 과도하게 장식적이지 않음

- 최소 본문 크기와 행간을 모바일에서 충분히 확보

- 커뮤니티
- SHOP
- 업체 광고
- 기업 서비스

노출 방식:

- 커뮤니티는 직업 페이지와 에피소드 페이지 안에서 연결
- SHOP은 “현직자가 사용하는 장비”, “이 에피소드에 나온 도구”처럼 문맥 안에서 노출
- 기업광고는 관련 직업·에피소드·공고 문맥에서 명확한 광고표시와 함께 노출
- 기업 서비스는 별도 B2B 진입점과 대시보드 제공
- 관리자 기능은 완전히 분리된 Admin Console 제공

기능이 성장해 실제 사용성이 검증되면
데이터를 근거로 내비게이션 승격을 검토합니다.

- 광고·협찬 문의

├── 실제 사용 장비

모든 Episode, Job Posting, Community Topic, Product, Company Campaign은
가능하면 하나 이상의 Occupation과 연결되어야 합니다.

Occupation taxonomy 요구사항:

- 직업 대분류, 중분류, 세부 직업

- slug
- 다국어 표시명 확장 가능
- 관련 직업
- 진입 경로 태그
- 근무환경 태그
- 업무 특성 태그
- 공개 상태
- 검수 상태
- 근거 출처와 갱신 시점

직업을 정확히 모르는 사용자도 다음 감각적 기준으로 탐색할 수 있어야 합니다.

- 사람을 상대하는 일
- 혼자 집중하는 일
- 몸을 쓰는 일
- 기술을 익히는 일
- 밤에 하는 일

12. JOBDAY SHOP 경험

- 직업별 소모품과 관리용품
- 현직자 리뷰와 사용조건
- 안전·호환·규격 정보

나쁜 노출:

- 직업과 무관한 인기 상품
- 광고비순 상품 배치
- 검증 없이 “현직자 추천” 표시
- 일반 쇼핑몰과 동일한 카테고리 확장
- 가격비교만 남는 범용 소비재
- 판매자가 직접 작성한 가짜 체험기

초기에는 외부 구매 연결과 제휴를 우선 검토하고,
반복 수요와 차별성이 검증된 카테고리만 직접 판매를 검토하십시오.

상품 데이터는 Occupation, Episode, Usage Context와 연결하십시오.
